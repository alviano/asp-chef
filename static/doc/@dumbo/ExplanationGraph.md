The **@dumbo/Explanation Graph** operation computes an explanation graph for the program stored in `__program__`, the answer set stored in `__answer_set__`, the Herbrand base stored in `__herbrand_base__`, and the query atoms stored in `__query__`.

The result is stored in predicate `__explanation__`.

§§§§

This operation provides a high-level justification for why specific facts are true (or false) within a given stable model. It is the heart of the "Explainable ASP" features in the Dumbo framework.

#### What is an Explanation Graph?

An explanation graph (also known as a justification) is a data structure that links a "query" atom to the rules and facts that support it.
- **Nodes**: Represent ground atoms or rules.
- **Edges**: Represent the "because of" relationships (e.g., Atom A is true because Rule R fired, and Rule R's body was satisfied by Atoms B and C).

#### Inputs Required

To function correctly, this operation typically expects a model containing:
1.  `__program__`: The ASP rules.
2.  `__answer_set__`: The stable model being explained.
3.  `__herbrand_base__`: The full set of possible ground atoms (often generated by the **Herbrand Base** operation).
4.  `__query__`: The specific atom(s) you want to explain.

#### How to use

1.  **Search Models**: Find a stable model for your program.
2.  **Wrap**: Wrap the model into `__answer_set__`.
3.  **Herbrand Base**: Compute the grounding universe.
4.  **Explanation Graph**: Run this operation to generate the graph structure.
5.  **Graph/TreeSpider**: Use a visualization ingredient to display the resulting `__explanation__` atoms.

#### Use Case: Debugging Incoherence or Surprising Results

When a solver returns a result you didn't expect, or when you need to provide a human-readable argument for an AI-generated decision, the **Explanation Graph** allows you to trace the logical derivation back to the initial input facts.
